# 2026-03-28 数据可视化导向后端建设与数据方案

## 1. 目标定位

本项目当前展示形态包含列表、详情、AI问答、3D浏览。结合赛道要求，核心定位应明确为：

1. 主体是古建筑信息的数据可视化与知识表达。
2. 3D模型是可视化载体，不是唯一核心功能。
3. 后端要从“内容分发接口”升级为“可视化数据服务层”。

这意味着后端不仅要返回建筑详情，还要提供可直接驱动地图、时间轴、统计图和讲解热点的数据聚合能力。

## 2. 当前后端能力与缺口

### 2.1 已具备能力

当前后端已经具备以下基础能力：

1. 建筑列表和详情。
2. 知识库问答匹配。
3. 素材查询。
4. 3D模型元数据与 manifest。
5. 3D资源签名与续期。
6. JSON/PostgreSQL 双数据源切换。

### 2.2 对“数据可视化”仍然缺失的能力

1. 缺少地图可视化所需经纬度、行政区、遗产等级等结构化字段。
2. 缺少时间轴可视化所需的统一年代区间与关键事件数据。
3. 缺少统计可视化的预聚合接口（按朝代、类别、地域、保护等级等）。
4. 缺少“数据来源与版权”元数据，无法做合规追溯。
5. 缺少“3D热点与知识点”统一映射模型，不利于可视化叙事。

## 3. 后端应优先做什么（按优先级）

## 3.1 P0（必须先做）

1. 数据模型补齐：围绕地图、时间轴、统计图补关键字段。
2. 新增可视化聚合接口：让前端不做重计算。
3. 数据来源追溯：每条关键数据都能追到来源和许可。
4. 数据质量校验：在 seed 前做字段完整性和枚举值校验。

## 3.2 P1（建议第二阶段）

1. 建立“知识点-建筑-3D热点”三者映射。
2. 提供叙事视图接口（例如“同类型建筑演化路线”）。
3. 增加接口缓存层（按查询维度缓存统计结果）。

## 3.3 P2（可选增强）

1. 建立简易数据后台（录入、审核、发布状态）。
2. 增加用户行为埋点（热点点击、3D停留时长、问答命中率）。
3. 做数据版本化（便于比赛后持续迭代）。

## 4. 建议新增的数据实体与字段

建议在 PostgreSQL 中新增或补齐以下实体。

### 4.1 buildings（已有表建议扩展）

新增字段建议：

1. `province`：省级行政区。
2. `city`：城市。
3. `lat`：纬度。
4. `lng`：经度。
5. `heritage_level`：文保等级（national/provincial/city/none）。
6. `open_status`：开放状态（open/partial/closed/unknown）。
7. `main_era_start`：主要建造年代起点（可用年份整数）。
8. `main_era_end`：主要建造年代终点。

### 4.2 building_events（新增）

用途：驱动时间轴与演化故事。

字段建议：

1. `id`
2. `building_id`
3. `event_year`
4. `event_type`（build/repair/damage/protection/listing）
5. `title`
6. `description`
7. `source_id`

### 4.3 building_metrics（新增）

用途：驱动对比类图表（结构、尺度、材料）。

字段建议：

1. `id`
2. `building_id`
3. `footprint_area_m2`
4. `height_m`
5. `structure_type`（wood/brick/stone/earth/mixed）
6. `primary_materials`（jsonb）
7. `protection_score`（0-100，内部评分可选）

### 4.4 data_sources（新增）

用途：数据可信与版权合规追溯。

字段建议：

1. `id`
2. `name`
3. `source_url`
4. `license_type`（cc0/cc-by/public-domain/custom）
5. `license_note`
6. `retrieved_at`
7. `proof_snapshot_url`（可选，证据留档）

### 4.5 building_hotspot_knowledge（新增）

用途：把 3D 热点与知识卡片绑定，支持“点哪里讲哪里”。

字段建议：

1. `id`
2. `building_id`
3. `model_version`
4. `hotspot_id`
5. `knowledge_id`
6. `priority`

## 5. 建议新增后端接口（面向可视化）

保持现有响应包结构：`{ code, msg, data }`。

1. `GET /api/visualization/overview`
   功能：返回首页看板核心指标（建筑总数、可看3D数、分类分布、朝代分布）。

2. `GET /api/visualization/map-points`
   功能：返回地图点位数据（id、name、category、lat、lng、heritageLevel、model3dStatus）。

3. `GET /api/visualization/timeline`
   功能：返回时间轴聚合数据，支持 `category`、`region`、`eraRange` 过滤。

4. `GET /api/visualization/stats`
   功能：返回图表数据集，支持 `dimension=category|era|region|heritageLevel`。

5. `GET /api/buildings/:id/insights`
   功能：返回单建筑“可视化讲解包”，包含事件线、结构指标、热点知识映射。

## 6. 需要找哪些数据

建议分为“核心必备”和“增强可选”两层。

### 6.1 核心必备（先满足比赛演示）

1. 建筑基础坐标：经纬度、省市、区域。
2. 时代信息：建造年代、主要修缮年份。
3. 类型信息：宫殿/桥梁/园林/城防/民居/楼阁/水利。
4. 文保信息：是否国保、省保、世界遗产相关标识。
5. 可视化叙事信息：3-5条关键事件，2-4个结构亮点。

### 6.2 增强可选（提升作品上限）

1. 尺度指标：面积、高度、跨度等。
2. 保护现状：开放状态、修缮状态、风险等级。
3. 关联知识图谱：人物、朝代、工艺、制度关系。

## 7. 数据来源建议（优先官方与可追溯）

建议优先“官方机构发布 + 开放许可明确 + 可留证据”的来源。

1. 国家文物局与地方文物部门公开名录。
2. UNESCO 世界遗产公开资料。
3. 各博物馆/景区官方公开资料页（需记录来源链接与抓取时间）。
4. 开放数据平台（如政府数据开放平台）中与文旅文保相关数据集。
5. 学术数据库或论文中的可引用事实数据（仅抽取可核验字段）。

说明：

1. 3D模型来源与知识数据来源要分开管理。
2. 任何非 CC0 或非明确许可资源，都要单独标注许可说明。
3. 比赛提交材料中建议附“数据来源清单表”。

## 8. 3D资源相关的后端职责（结合你给的方案）

你整理的“免费模型获取 + 转GLB”方案可作为素材生产流程，但后端需要补上这三层能力：

1. 资源元数据管理：模型来源、许可、版本、面数、文件体积、压缩方式。
2. 资源发布治理：`draft -> review -> published` 状态机，避免未审资源上线。
3. 资源可用性约束：对外只给签名地址，不透传裸链路。

建议在 `building_profiles.model3d.versions[]` 旁补充以下字段：

1. `sourceName`
2. `sourceUrl`
3. `licenseType`
4. `polyCount`
5. `fileSizeMb`
6. `lastVerifiedAt`

## 9. 实施节奏建议（7天可落地）

1. Day 1：确定字段字典与枚举，冻结数据标准。
2. Day 2：补 migration，创建 `building_events`、`building_metrics`、`data_sources` 等表。
3. Day 3：补 seed 脚本，先覆盖 17 个建筑的核心字段。
4. Day 4：实现 `/api/visualization/overview` 与 `/api/visualization/map-points`。
5. Day 5：实现 `/api/visualization/timeline` 与 `/api/visualization/stats`。
6. Day 6：实现 `/api/buildings/:id/insights`，打通 3D 热点映射。
7. Day 7：联调前端图表与地图页面，补数据来源清单与验收文档。

## 10. 验收标准

1. 地图页不再依赖前端硬编码分类统计，全部来自后端聚合接口。
2. 时间轴和统计图不依赖前端拼接文本，全部由结构化数据驱动。
3. 任一建筑都可追溯数据来源与许可信息。
4. 任一3D版本都可追溯来源、许可与发布时间。
5. 前端可在 2 秒内拿到首页可视化核心数据（开发环境可适当放宽）。

## 11. 对当前项目的直接建议

结合当前代码状态，建议优先做下面四件事：

1. 在 `buildings` 数据中先补 `lat/lng/province/city/main_era_start/main_era_end`。
2. 新增 `visualizationService` 与 `visualizationController`，先做 overview + mapPoints 两个接口。
3. 给 `building_profiles.model3d.versions[]` 增加来源与许可字段，避免后续版权风险。
4. 新增 `docs/数据来源清单.md`，沉淀每条关键数据的来源证据。

以上四项完成后，你的后端就从“展示型 API”进入“数据可视化平台 API”阶段，后续加图表和地图会非常顺滑。
