# 互联网数据采集与供数执行手册

## 1. 文档定位

本手册用于指导从互联网找数据的同学开展标准作业，目标是把外部数据稳定转成可入库、可检索、可供前端调用的结构化数据。

适用范围：

1. 古建筑基础信息采集。
2. 可视化数据供数（信息图、图表、地图、3D 配置）。
3. 知识问答数据供数（knowledge_base 与向量化）。

---

## 2. 岗位命名（你问的叫啥）

建议统一叫：

1. 数据采集与供数工程师（推荐）
2. 数据采集工程师（简写）

团队内简称建议：采数同学。

---

## 3. 工作目标

采数同学要完成三件事：

1. 从互联网采集可信数据。
2. 按项目字段标准清洗加工并入库。
3. 输出给前端可直接消费的接口数据，不让前端自己猜字段。

---

## 4. 端到端流程（必须按顺序）

1. 明确采集任务：本次要补哪一类数据（建筑基础/时间轴/地图/3D/问答）。
2. 采集原始资料：记录来源 URL、抓取时间、版权许可。
3. 清洗与标准化：统一命名、单位、年份、地理字段。
4. 字段映射：映射到项目标准表结构。
5. 入库：先本地验证，再写入目标库（测试或生产）。
6. 回归检查：跑接口检查与检索回归。
7. 交付：给前端发布可用数据批次说明。

---

## 5. 数据来源规则（互联网采集）

## 5.1 来源优先级

1. 官方来源（博物馆官网、文旅局、政府公开数据）
2. 权威机构来源（高校、研究机构、出版数据库）
3. 商业平台公开信息（仅作补充，需二次核验）

## 5.2 必须记录的元信息

每条采集数据都要记录：

1. sourceUrl
2. sourceName
3. collectedAt
4. licenseNote（转载许可或使用限制）
5. confidence（low/medium/high）

## 5.3 合规要求

1. 不抓取需要登录且无授权的数据。
2. 不直接复制受限版权全文到知识库。
3. 对争议信息要加 confidence=low 并标注来源。

---

## 6. 项目字段标准（入库前必须对齐）

## 6.1 建筑基础数据（buildings）

关键字段：

1. id（唯一，建议 snake_case）
2. name
3. category（palace|bridge|garden|defense|residence|tower|water）
4. location
5. province
6. city
7. lat
8. lng
9. heritage_level
10. open_status
11. main_era_start
12. main_era_end
13. description
14. image
15. tags

## 6.2 建筑扩展档案（building_profiles + model_*）

关键字段：

1. overview_summary
2. history
3. cultural_value
4. key_points
5. architecture_highlights
6. model3d_status
7. model3d_viewer_type
8. model3d_poster
9. model_versions / lod / hotspots

## 6.3 问答数据（knowledge_base）

关键字段：

1. question
2. answer
3. material_id
4. keywords
5. category（可选）

规则：

1. question 要贴近用户自然问法。
2. answer 保持事实准确、长度适中。
3. keywords 覆盖别名与常见口语。

## 6.4 向量数据（knowledge_embeddings）

由脚本生成，不人工维护 embedding 数组。

关键字段：

1. knowledge_id
2. embedding_model
3. embedding_dim
4. embedding
5. question_text

---

## 7. 四类可视化供数映射（基于 temp 要求）

## 7.1 信息图（Info-graphic）

前端接口：

1. GET /api/visual/info-graphic
2. GET /api/visualization/timeline

供数字段：

1. type, title, layout
2. nodes, edges, labels, links

## 7.2 MG 动画

前端接口：

1. 预留 GET /api/visual/mg-animation

供数字段：

1. animationId, title
2. duration, lottieUrl
3. autoPlay, loop
4. keyPoints

## 7.3 3D 交互

前端接口：

1. GET /api/buildings/:id/model3d
2. GET /api/buildings/:id/model3d/manifest

供数字段：

1. modelUrl 或 scene.entry
2. camera / initConfig
3. markers 或 hotspots
4. tokenPolicy

## 7.4 图表（Chart）

前端接口：

1. GET /api/visual/chart
2. GET /api/visualization/stats

供数字段：

1. title, unit
2. xAxis
3. series
4. legend
5. tooltip

---

## 8. 入库策略（本地与 VPS）

## 8.1 推荐策略

1. 结构迁移在 VPS 跑（db:migrate）。
2. 初始化数据在 VPS seed（db:seed）。
3. 生产新增数据以 VPS 为主，不反复用本地覆盖。

## 8.2 采数同学日常节奏

1. 本地先整理成标准 JSON/CSV。
2. 在测试库导入并验证。
3. 通过后合并到正式批次，在 VPS 执行入库。

---

## 9. 验收标准（采数同学必须自检）

## 9.1 数据质量自检

1. 空值率：关键字段不能为空（name/category/province/city/year）。
2. 地图字段：lat/lng 必须为合法数字。
3. 时间字段：main_era_start <= main_era_end（或 end 为空）。
4. 分类字段：必须在枚举值内。

## 9.2 接口回归自检

1. /api/visualization/overview 返回 summary、distributions、kpis。
2. /api/visualization/map-points 返回 points、markers、geoJson。
3. /api/visualization/timeline 返回 timeline、infographic。
4. /api/visualization/stats 返回 items、chart。
5. /api/chat 返回 source 与 materialId，开发环境可见 debug。

## 9.3 检索回归自检

执行：

1. npm run test:chat-regression
2. npm run scan:chat-thresholds

目标：

1. 错别字/别名样本通过。
2. 无关问题不误召回。

---

## 10. 采数交付模板（给后端/前端）

每个数据批次交付时，必须附带：

1. 批次编号：data-batch-YYYYMMDD-N
2. 数据范围：建筑数、知识条数、3D 资源数
3. 来源列表：sourceName + sourceUrl
4. 字段变更：新增/修改/删除
5. 验证结果：接口回归结果、检索回归结果
6. 风险备注：争议数据、低置信度数据

---

## 11. 常见问题

## 11.1 切到 postgres 启动报字段不存在

处理：

1. npm run db:migrate
2. 检查 DATABASE_URL 是否指向正确库

## 11.2 向量检索命中差

处理：

1. 先补高质量 question/keywords。
2. 再调阈值，不要只调阈值。

## 11.3 可视化字段对不上

处理：

1. 以后端接口总览与对接手册为准。
2. 变更字段必须先加后删，给前端过渡期。

---

## 12. 与前端协作约定

给前端统一口径：

1. 图表/地图/时间轴只消费结构化接口数据。
2. Chat 文本不作为可视化主数据源。
3. 字段变更提前通知，附示例响应。
