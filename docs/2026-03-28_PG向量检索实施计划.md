# PG 向量检索实施计划（关键词优先）

## 1. 目标

在不破坏现有接口的前提下，为 `/api/chat` 增加“语义检索”能力，并保持你当前的优先级：

1. 关键词精确匹配（第一优先，快）
2. 向量语义检索（第二优先，稳）
3. 大模型自由回答（最后兜底）

目标效果：

1. 用户复杂问法也能命中知识库。
2. 返回结构保持兼容：`answer + source + materialId`。
3. 可以灰度开关，出现问题可一键回退到当前逻辑。

## 2. 当前现状（已确认）

当前后端链路：

1. `chatService.getChatResponse(question)` 先调用 `findBestKnowledgeMatch`。
2. 目前 `knowledgeService` 使用“文本包含 + 关键词包含”打分。
3. 分数不足阈值后，直接调用大模型回答。

这说明你的架构方向已经正确，只需在“匹配不到时”插入语义检索层。

## 3. 最终流程（建议）

```text
用户提问
  -> Step A: 关键词匹配（当前能力，先增强）
  -> 命中: 返回 knowledge_base
  -> 未命中: Step B: 向量检索（PG + pgvector）
  -> 命中: 返回 knowledge_vector
  -> 未命中: Step C: 大模型兜底回答
```

建议命中阈值：

1. 关键词匹配阈值：`>= 30`（可配置）
2. 向量检索阈值：`similarity >= 0.78`（先经验值，后面按日志调）

## 4. 分阶段实施

### Phase A（现在就能做，不依赖 pgvector）

目标：先把“关键词检索”做到可上线可观测。

任务：

1. 提取可配置参数（阈值、权重）到环境变量。
2. 增加同义词/别名归一化（如“故宫最大那个殿” -> “太和殿”候选）。
3. 输出检索调试日志（仅开发环境）：命中项、分数、来源。
4. 给 `/api/chat` 响应增加可选调试字段（仅 dev，默认关闭）。

产出：

1. 一版可解释、可调参的关键词检索。
2. 为接入向量检索打好评估基线（命中率、兜底率）。

### Phase B（pgvector 环境准备）

目标：让 PostgreSQL 具备向量能力。

任务：

1. 安装并启用扩展：`CREATE EXTENSION IF NOT EXISTS vector;`
2. 新增迁移脚本（建议 `003_add_vector_retrieval.sql`）。
3. 新建向量表：`knowledge_embeddings`。
4. 为向量列建立 ivfflat 索引。

建议表结构：

1. `id` serial PK
2. `knowledge_id` int FK -> `knowledge_base.id`
3. `embedding_model` varchar(100)
4. `embedding_dim` int
5. `embedding` vector(维度)
6. `question_text` text
7. `metadata` jsonb
8. `created_at / updated_at`

索引建议：

1. `knowledge_id` BTree
2. `embedding vector_cosine_ops` + ivfflat

### Phase C（向量入库与检索）

目标：把知识库问题向量化并可检索。

任务：

1. 新增 embedding 服务（支持豆包/智谱/文心三选一）。
2. 新增脚本：全量向量化 + 增量向量化。
3. 生成用户问题向量，执行 TopK 相似检索。
4. 基于阈值判断是否命中。

SQL 检索示例（余弦距离）：

```sql
SELECT
  ke.knowledge_id,
  1 - (ke.embedding <=> $1::vector) AS similarity
FROM knowledge_embeddings ke
ORDER BY ke.embedding <=> $1::vector
LIMIT 5;
```

### Phase D（接入聊天主链路）

目标：完成“关键词 -> 向量 -> AI”完整闭环。

任务：

1. 在 `chatService` 中插入向量检索步骤。
2. `source` 增加新值：`knowledge_vector`。
3. 增加开关：`ENABLE_VECTOR_RETRIEVAL=true|false`。
4. 增加回滚策略：关闭开关即恢复旧链路。

## 6. 你现在需要准备的东西（先准备，不阻塞开发）

### 必备

1. PostgreSQL 版本确认（建议 15+）。
2. pgvector 安装方式确认（本机安装或 Docker）。
3. 选一个 Embedding 提供方（豆包 / 智谱 / 文心）。
4. 准备对应 API Key（测试环境即可）。

### 建议提前确定

1. 向量维度（由所选模型决定，如 768/1024/1536）。
2. 成本策略：
   - 离线全量向量化频率（每天/手动）
   - 增量更新触发（新增知识时）
3. 可接受响应时间目标（例如 P95 < 800ms）。

## 7. 环境变量建议

```env
# 向量检索开关
ENABLE_VECTOR_RETRIEVAL=false

# 检索参数
CHAT_KEYWORD_SCORE_THRESHOLD=30
CHAT_VECTOR_SIMILARITY_THRESHOLD=0.78
CHAT_VECTOR_TOP_K=5

# Embedding
EMBEDDING_PROVIDER=doubao
EMBEDDING_API_KEY=your_key
EMBEDDING_MODEL=your_embedding_model
EMBEDDING_DIM=1024
```

## 8. 验收标准

1. 关键词问题命中率不低于当前版本。
2. 复杂自然语言问题命中率显著提升（建议挑 30 条测试样本）。
3. `/api/chat` 响应结构保持兼容。
4. 关闭 `ENABLE_VECTOR_RETRIEVAL` 后可立即回退到旧逻辑。
5. 出错时不影响主链路：向量检索异常自动降级到 AI 兜底。

## 9. 风险与规避

1. 风险：向量阈值设太高导致经常走 AI。
   - 规避：先记录 TopK 分数分布，再调阈值。
2. 风险：Embedding 模型切换导致向量不兼容。
   - 规避：表中记录 `embedding_model` 与 `embedding_dim`，分批重建。
3. 风险：向量检索慢。
   - 规避：先小数据量验证，再加 ivfflat 并调 `lists` 参数。

3. 环境就绪后进入 Phase B + C。
4. 最后做 Phase D 联调与灰度。
